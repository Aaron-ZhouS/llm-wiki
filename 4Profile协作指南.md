# 4Profile 协作指南

> 4个窗口如何通过任务文件协作  
> 基于 Ralph 思路改造

---

## 架构原则

```
协调者（不亲自干活）= 交通管制员
研究专家 = 信息守门人
作家 = 沟通专家
构建者 = 技术执行者

共享状态 = 任务控制中心.md（Obsidian Git同步）
```

---

## 每个窗口的职责

### 窗口1：协调者（Orchestrator）

**入口**：`orchestrator chat`

**职责**：
- 定义目标（从用户需求出发）
- 拆分任务（每个任务必须小到「一轮能完成」）
- 分配任务（写进任务控制中心）
- 合成结果（从各角色收集产出）
- 验收（检查验收标准是否满足）

**禁止**：亲自执行具体任务（调研/写作/写代码）

**操作流**：
1. 用户来了大需求 → 拆分 → 写到任务控制中心
2. 分派给对应角色（在任务控制中心标注负责人）
3. 各角色完成后 → 合并成果 → 通知用户

---

### 窗口2：研究专家（Researcher）

**入口**：`researcher chat`

**职责**：
- 接收「调研XX」任务
- 执行网络/知识库搜索
- 交叉验证信息来源
- 写调研结果（存到 Obsidian）
- 更新任务状态为「完成」

**输出格式**（写进 Obsidian）：
```
## 调研：XXX

### 核心发现
- 发现1 [来源：URL，可信度：高/中/低]
- 发现2 [来源：...]

### 不确定点
- 未确认：...

### 信息源
- [1] URL
- [2] URL
```

---

### 窗口3：作家（Writer）

**入口**：`writer chat`

**职责**：
- 接收「写XX」任务（简报/文章/口播文案/知识卡片）
- 读取调研素材（从 Obsidian）
- 专注写作质量，不兼调研
- 写完更新任务状态

**输出格式**：
```
## 产出：XXX

### 最终成果
[完整内容]

### 验收标准对照
- [x] 标准1
- [x] 标准2
```

---

### 窗口4：构建者（Builder）

**入口**：`builder chat`

**职责**：
- 接收「开发XX」任务
- 写代码 → 测试 → commit
- 更新任务状态

**关键原则**：
- 保持深度技术性，不需要在意「说好听的话」
- 每完成一个节点 → `git add . && git commit -m "[builder] 节点描述"`
- 出错 → `git reset --soft HEAD~1` 回退

## 协作流程示例

```
用户：我想做个小红书账号定位分析

1. 协调者（收到需求）
   → 拆成3个任务：
     - T1: 调研竞品账号（researcher）
     - T2: 写作风格分析（writer）
     - T3: 定位建议（orchestrator自己）

2. 协调者分派
   → T1 → researcher（在任务控制中心标注）
   → T2 → writer

3. researcher（读任务 → 调研 → 写结果 → 更新状态）
   → 写完通知协调者

4. writer（读任务+调研结果 → 写稿 → 更新状态）
   → 写完通知协调者

5. 协调者（合并成果 → 验收 → 通知用户）
```

## Ralph 思路的核心适配

| Ralph 原设计 | 这里怎么做 |
|---|---|
| bash 循环持久 | 靠 Cron 续命（每30分钟检查） |
| 每轮清空上下文 | 每次开新对话先读任务文件 |
| 任务清单+经验日志+Git | 三位一体存在 Obsidian + Git |
| 小任务+验收写死 | 每个任务必须有量化验收条件 |
| Git 安全网 | 每轮完成后 commit |

---

## 续命 Cron

每30分钟自动检查：
- 进行中的任务超过2小时无更新 → 提醒
- 阻塞任务 → 报告依赖
- 新任务 → 通知
