# 拆书 → 拍摄知识块 → 内化 v2

> **核心理念**：把别人的书拆成自己的东西。三段式：先客观拆骨架，再推荐可拍内容，最后内化进知识库。**开头不加废话，直接出结果。**

---

## 一句话定位

**输入（一段书/文章/书名/URL）→ 5 段客观提取 → 2-4 条可拍知识块 → 内化进知识库。**

---

## 3 段流程（核心方法）

### Step 0：输入检测

| 输入形态 | 怎么识别 | 动作 |
|---|---|---|
| 已含正文 | 用户直接粘贴文本 | 跳过读取，直接处理 |
| 文件路径 | `/` 或 `~` 开头，扩展名 `.md/.pdf/.txt/.epub` | 用 `read_file` 读取 |
| URL | `http://` 或 `https://` 开头 | 用 `web_fetch` / `web_search` 拉取 |
| 仅有书名 | 没有上述任何一种 | 让用户粘一段内容或给个文件/链接 |
| 仅有触发词 | 用户只打了 `/拆书` / `/拆书博主`，没带内容 | 回问一次："把要拆的贴上来——文本/文件路径/链接，三选一" |

**反问话术规则**：
- 只问一次，不连环追问
- 给出三选一选项，降低用户决策成本
- 用户补完内容后按 Step 0 → Step 1 顺序执行

### Step 1：客观提取（5 段固定输出，缺一不可）

```
📖 作者核心观点    概括 3-5 个核心观点，每个一句话
📝 关键原文        摘录 3-5 段最关键的原文，逐字引用
💡 洞见            提取 2-4 条可独立传播的钉子句 + 支撑 + 原文出处
🪞 个人映射        仅当输入含明确人名/身份信息时输出（直接关联 + 反直觉冲击 + 可执行方向）
🎬 案例/故事       作者用什么具体场景在论证
⚡ 一句话带走      这本书如果只能留一句话是什么
```

### Step 2：推荐拍摄知识块（从洞见中筛 2-4 条）

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎬 推荐拍摄 1/3
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎣 hook（钉子句）
   一句话，让人想截图的那种。
🔥 为什么值得讲
   这个观点厉害在哪里？反常识的点是什么？
😖 大众痛点
   普通人在这件事上的常见误区/焦虑是什么？具体场景化。
```

（完整模板见 `templates/shoot-block.md`）

### Step 3：内化进知识库（可选）

把拆出的洞见写入知识库，关联到现有概念：
- 单条洞见 → `concepts/xxx-洞见.md`
- 整本书总结 → `concepts/xxx-书评.md`
- 跨书主题 → 主题文件夹

## 5 段固定输出详解

### ① 作者核心观点
- 数量：3-5 个
- 形式：每条一句话
- 标准：能让没读过书的人知道作者在讲什么

### ② 关键原文
- 数量：3-5 段
- 形式：逐字引用
- 标准：选最有冲击力/最反常识/最实用的段落

### ③ 洞见（钉子句）
- 数量：2-4 条
- 形式：钉子句 + 支撑 + 原文出处
- 标准：可以独立传播的"可截图"金句

### ④ 个人映射
- 触发：仅当输入含明确人名/身份信息时
- 形式：直接关联 + 反直觉冲击 + 可执行方向
- 标准：让用户立刻看到"这跟我有什么关系"

### ⑤ 案例/故事
- 数量：1-3 个
- 形式：作者用的具体场景
- 标准：有画面感、能让读者"看到"这件事

### ⑥ 一句话带走
- 数量：1 句
- 形式：30 字以内
- 标准：如果只能留一句话，就是这一句

## 何时用 book-extract

**用 book-extract 的信号**：
- 有一本书/文章想拆成可拍内容
- 想快速提取一本书的精华
- 想做读书博主/拆书博主
- 想把别人的方法论内化进自己的知识库

**不用 book-extract 的信号**：
- 只需要一句话的总结（用 linggan-alchemy）
- 没看完书就想拆（先读完）
- 想自己写原创（不是拆书，是创作）
- 想要 100% 客观分析（用 topical-reading-ai）

---

## 红灯边界（硬门禁）

- 🚫 用户只给书名没给内容——回问一次，不要瞎猜
- 🚫 拆的内容不真实——只引用原文，不编造
- 🚫 钉子句太多——≤4 条，超出就合并
- 🚫 把别人的话当成自己的——必须标出处
- 🚫 内容来源是盗版/未授权——拒绝拆

**拒绝话术**：
> "我需要看到原书内容（贴文本/给文件/给链接），不能凭空拆。"

---

## 检查点设计

| 检查点 | 触发 | 用户需要确认 |
|--------|------|-------------|
| **CP1** | Step 0 后 | 输入形式识别正确吗？ |
| **CP2** | Step 1 后 | 5 段输出都齐了吗？钉子句 ≤4 条吗？ |
| **CP3** | Step 2 后 | 2-4 条拍摄块都有 hook/痛点/价值吗？ |
| **CP4** | Step 3 后 | 是否要内化进知识库？关联到哪个概念？ |

---

## 失败处理（Fallback）

| 失败场景 | Fallback |
|---------|---------|
| 用户只给书名 | 回问一次"贴一段内容/给文件/给链接" |
| 书太短拆不出 5 段 | 降到 3 段，缺哪段标"无" |
| 没有洞见 | 改用 5WHY 提问挖洞见 |
| 用户没读完 | 提醒"先读完再拆，否则质量差" |
| 来源不可靠 | 拒绝拆，建议找其他来源 |
